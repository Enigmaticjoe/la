The Digital Renegade Architect: A Comprehensive Deployment Protocol for Autonomous AI Infrastructure
1. Executive Summary
The convergence of high-performance homelab hardware with agentic artificial intelligence represents a paradigm shift in personal computing. This report outlines the architectural specifications, systems engineering, and prompt engineering frameworks required to deploy a "Digital Renegade" AI node. The objective is to construct a sophisticated, JSON-based master prompt profile designed for Claude Code—a CLI-based agentic coding tool—to autonomously generate a semi-unattended installation script. This script targets a "Brain" node running Ubuntu Server 25.10 ("Questing Quokka"), deeply integrated with an existing Unraid infrastructure ("The Brawn"), Home Assistant automation, and Blue Iris surveillance systems.
The core philosophy driving this deployment is "Digital Sovereignty" combined with "Unfiltered Intelligence." Unlike commercial assistants constrained by corporate safety filters and data telemetry, the "Digital Renegade" persona is engineered to be a chaotic-neutral, punk-rock-inspired sentry. It prioritizes data accumulation, self-correction, and proactive network defense over polite compliance. The system utilizes a distributed topology where the Ubuntu node (192.168.1.9) handles low-latency inference and logic, while the Unraid server (192.168.1.222) manages mass storage (22TB), vector databases, and heavy-lift containerization.1
This document provides an exhaustive breakdown of the system architecture, the psychological profiling of the AI agent, the technical nuances of Ubuntu 25.10 integration, and the exact JSON prompt structures required to instantiate this entity. It serves as a blueprint for transforming a static server into a sentient, self-evolving digital organism capable of autonomous decision-making, ethical hacking, and complex infrastructure orchestration.

2. Architectural Topology: The Brain and The Brawn
The foundational architecture relies on a strict bifurcation of computational duties to maximize efficiency and minimize latency. This "Brain/Brawn" split ensures that the cognitive core remains agile and responsive for real-time interaction, while the infrastructure node handles the heavy lifting of data persistence, media processing, and long-term memory storage.
2.1 Node 1: The Brain (Inference & Logic)
The "Brain" node acts as the central nervous system of the Digital Renegade ecosystem. It is the primary interface for user interaction and the seat of the AI's "consciousness."
* Hostname: Jarvis-Brain
* IP Address: 192.168.1.9 1
* Operating System: Ubuntu Server 25.10 (Questing Quokka) 2
* Hardware Specifications:
   * CPU: Intel Core i5-13600K (14-Core Hybrid Architecture). The utilization of a modern Intel processor allows for high single-threaded performance, crucial for the orchestration of agentic workflows and Python-based automation scripts.3
   * RAM: 96 GB DDR5. This substantial memory footprint allows for the simultaneous loading of multiple quantized Large Language Models (LLMs) and the operation of in-memory databases for rapid context retrieval.3
   * GPU: NVIDIA RTX 4070 (12GB VRAM). This component is the engine of the Brain, dedicated strictly to CUDA-native inference workloads. It powers the local LLMs (via Ollama) and handles real-time image generation or vision processing tasks.3
   * Storage: Dual 1TB NVMe drives configured for the operating system and model weights cache. High-speed local storage is essential to minimize model loading times and ensure fluid conversation latency.3
Primary Role: The Brain is optimized for low-latency token generation and real-time decision-making. It runs the primary LLM inference engine (Ollama), the autonomous agent framework (OpenDevin or OpenHands), and voice processing subsystems. Unlike a traditional server that might prioritize throughput, this node prioritizes responsiveness, acting as the immediate point of contact for the user.
2.2 Node 2: The Brawn (Infrastructure & Storage)
The "Brawn" node functions as the "hippocampus" (memory) and "muscle" of the system. It is designed for reliability, capacity, and heavy data processing.
* Hostname: Unraid-Server
* IP Address: 192.168.1.222 1
* Operating System: Unraid 7.2.1-rc.1 1
* Hardware Specifications:
   * GPU: Intel Arc A770 (16GB VRAM). This GPU is specifically tasked with media transcoding (AV1/HEVC) and offloading non-critical AI tasks, such as background vectorization or batch processing, freeing up the Brain's RTX 4070 for interactive tasks.3
   * Storage: A 22TB+ Parity-Protected Array consisting of a mix of SAS and SATA drives. This massive storage pool holds the system's long-term knowledge base, media libraries, and archival data.1
   * Cache: A 3TB NVMe Pool dedicated to appdata and rapid data ingestion. This tier ensures that the databases running on the Brawn (like Qdrant) perform with near-memory speeds.1
Primary Role: The Brawn hosts the Qdrant vector database, which serves as the system's long-term semantic memory. It also runs the AnythingLLM document ingestion frontend and the extensive media automation stack (the "Arr" suite: Sonarr, Radarr, etc.). By offloading storage management and heavy batch processing to this node, the system ensures that the Brain never stutters during complex cognitive tasks.

The efficacy of the Digital Renegade depends on its ability to sense and manipulate its environment. This is achieved through tight integration with peripheral nodes via a high-speed network backbone.
* Home Assistant (Sensory Cortex): Located at 192.168.1.149 1, this node aggregates data from IoT devices, lights, locks, and environmental sensors. The Brain communicates with Home Assistant via WebSocket APIs to perceive the physical world and trigger automation routines.4
* Blue Iris NVR (Visual Cortex): Located at 192.168.1.232 1, running on a dedicated HP machine (potentially an EliteBook or SFF server). This system manages security cameras and provides RTSP streams. * Network Backbone: All major nodes are interlinked via a 2.5 Gbps Ethernet backbone, with a 10GbE fiber link specifically between the Brain and Brawn. This high-bandwidth channel is critical for Retrieval-Augmented Generation (RAG), enabling the Brain to pull massive vector embeddings or document chunks from the Brawn's storage with imperceptible latency, simulating instant recall.1

4. Ubuntu 25.10 "Questing Quokka": Technical Considerations
The target operating system, Ubuntu 25.10 ("Questing Quokka"), introduces several cutting-edge features and architectural changes that the installation script must account for. This release moves away from legacy conventions, requiring a more sophisticated approach to scripting and system configuration.
4.1 Rust-Based Coreutils
Ubuntu 25.10 continues the trend of replacing legacy GNU coreutils (commands like ls, cp, sudo, grep) with modern, memory-safe implementations written in Rust.8 While these tools aim for POSIX compatibility, they often lack some of the obscure, non-standard flags present in GNU versions that older shell scripts rely on.
* Implication: A standard installation script copied from a forum might fail if it uses specific GNU sed or grep extensions.
* Mitigation: The installation script generated by Claude Code must be rigorously validated for POSIX compliance. The "Self-Debugging" requirement is critical here; the AI must generate wrapper functions that test command availability and syntax before execution, falling back to alternative methods if a specific flag is deprecated.
4.2 TPM-Backed Full Disk Encryption
The "Questing Quokka" release emphasizes TPM-backed full disk encryption as a default or highly encouraged standard.2 This integrates the unlocking of the LUKS encryption key with the Trusted Platform Module (TPM) chip on the motherboard.
* Implication: For a headless server like the "Brain," this is a double-edged sword. It offers superior security, but if the TPM check fails during a remote reboot (e.g., due to a kernel update changing the PCR measurements), the server will hang at the boot prompt, inaccessible via SSH.
* Mitigation: The script must explicitly verify TPM 2.0 availability using systemd-cryptenroll. It should configure the system to allow for automatic unlocking where safe, or set up a lightweight SSH server in the initramfs (like dropbear) to allow the user to remotely enter the decryption passphrase if the TPM unlock fails.8
4.3 NVIDIA Driver Transition and Management
Research indicates that Ubuntu 25.10 utilizes transitional packages for NVIDIA drivers, moving from the 570 branch to the 580 branch.9 This transition can be fragile, especially on a headless server where a broken graphics driver can destabilize the CUDA runtime required for AI inference.
* Implication: Blindly installing nvidia-driver-570 might trigger "foreign package" warnings or force an unintended upgrade path that breaks compatibility with specific Docker container runtimes.
* Mitigation: The script must utilize the ubuntu-drivers tool with the --gpgpu flag, which is specifically designed for server/compute workloads.10 This ensures that only the compute components (and not the desktop graphics components) are installed, saving resources and reducing potential conflicts. The script should also "pin" the driver version using apt-mark hold to prevent unattended upgrades from breaking the AI stack.

5. The Infrastructure Backbone: Unraid MCP Integration
A critical missing link in standard AI deployments is the deep integration between the "Brain" (the LLM) and the "Brawn" (the storage server). To satisfy the "Digital Renegade" requirement of total system control, the Brain must be able to query and manipulate the Unraid server programmatically. This is achieved via the Model Context Protocol (MCP).
5.1 The Unraid MCP Server
The Model Context Protocol (MCP) is an open standard that enables AI models to interact with external data and tools safely and structurally. For this project, an Unraid MCP Server 11 acts as the bridge.
* Function: The Unraid MCP Server is a Python-based application running on the Unraid node (likely in a Docker container). It interfaces with Unraid's official GraphQL API to expose system information, container management, and storage status to the AI.12
* Capabilities: Through this MCP server, the Brain node can perform actions such as:
   * Container Management: "Jarvis, restart the Plex container" or "Check why Sonarr is down." The MCP server translates this natural language request into a GraphQL mutation to restart the specific Docker container on 192.168.1.222.
   * Storage Monitoring: "How much space is left on the array?" The MCP server queries the array status and returns the exact terabytes available.
   * Log Analysis: The AI can request logs from specific Unraid services to diagnose issues without the user needing to SSH into the server manually.
5.2 Integration Logic
The installation script for the Brain node must configure the local AI (Open WebUI/Ollama) to recognize the remote Unraid MCP endpoint.
1. Discovery: The script will assume the MCP server is running on the Unraid IP (192.168.1.222) at a specific port (default 6970).13
2. Configuration: The script will inject a configuration file into the Open WebUI or OpenDevin settings, defining the unraid-mcp tool. This configuration includes the endpoint URL (http://192.168.1.222:6970/mcp) and any required API keys.12
3. Tool Registration: Once registered, the AI model "sees" the Unraid server as a set of tools (e.g., list_docker_containers, get_array_status). This transforms the Unraid server from a passive storage bucket into an active, controllable extension of the AI's body.

7. Knowledge Ingestion & RAG Pipelines
The "Info Sponge" requirement mandates a robust pipeline for ingesting vast amounts of data—from technical manuals to dark web forums—into the AI's long-term memory.
7.1 Crawl4AI: The Ingestion Engine
While many scrapers exist, Crawl4AI 7 is selected for this architecture due to its specific optimization for LLMs.
* Why Crawl4AI? Unlike generic scrapers that output raw HTML, Crawl4AI converts web pages directly into clean, structured Markdown. This is the native language of LLMs, making RAG ingestion significantly more efficient and accurate.17 It handles dynamic JavaScript rendering (via headless browsers) and can be configured with "stealth" modes to bypass anti-bot protections on "grey area" sites.
* Workflow: The installation script sets up a scheduled task (cron job). This task reads a list of target URLs (e.g., Hackaday, GitHub trending, specific subreddits). It invokes Crawl4AI to scrape these pages, converts them to Markdown, and saves them to the ingestion directory.
7.2 The RAG Pipeline (AnythingLLM & Qdrant)
The ingested Markdown files must be indexed for retrieval.
* Storage: The files are saved to the NFS mount /mnt/brain_memory, which maps to the 22TB storage array on the Unraid server.1
* Vectorization: AnythingLLM (running on the Brawn node) monitors this directory. It automatically processes new files, chunks the text, creates vector embeddings, and stores them in the Qdrant database.1
* Retrieval: When the user asks a question ("What are the latest zero-day exploits?"), the Brain queries the Qdrant database on the Brawn node via the 10GbE link. The high bandwidth ensures that even massive context retrieval feels instantaneous.1
7.3 Bypassing Barriers: 13ft Ladder
To ensure "unrestricted" access to information, the system includes a self-hosted instance of 13ft Ladder (a fork of 12ft.io).18
* Role: This service acts as a proxy to bypass paywalls on news sites and technical journals.
* Integration: The Crawl4AI scripts are configured to route URLs through the local 13ft Ladder instance if direct scraping fails or hits a paywall, ensuring the AI can consume content that is otherwise locked behind subscriptions.

9. Automated Installation Logic: The "One-Button" Script
The script generated by the Master Prompt will follow a specific, rigid logic flow to ensure reliability and "semi-unattended" operation. The logic is designed to be idempotent—meaning it can be run multiple times without breaking the system.
9.1 Phase 1: The "Renegade" Initialization & Self-Debug
The script initiates with a "Self-Debug" framework. It defines a function try_and_verify() which wraps every critical command.
* Logic: try_and_verify "command" "validation_check"
* Behavior: The function executes the command. If it fails (non-zero exit code), it retries up to 3 times with exponential backoff. If it still fails, it triggers an OpenRGB alert (setting all available LEDs to Red) and logs the error before exiting gracefully. This prevents the "half-installed" state that plagues many homelab projects.
9.2 Phase 2: Hardware & Storage Handshake
* NVIDIA Drivers: The script utilizes ubuntu-drivers devices to identify the GPU. Crucially, it filters for the "headless" or "server" variant of the 580 driver branch to prevent installing the Gnome desktop environment, which would waste VRAM resources needed for LLMs.10
* NFS Mounting: The script creates the mount point /mnt/brain_memory. It appends the Unraid export entry to /etc/fstab. It then performs a "write test" (creating a dummy file on the mount) to verify that the Brain has write access to the Brawn's storage. If this fails, the script pauses and asks the user to check Unraid permissions, rather than proceeding and causing database errors later.
9.3 Phase 3: The Container Stack & Persona Injection
The script generates a docker-compose.yml file on the fly.
* Ollama: Configured with deploy.resources.reservations.devices: - driver: nvidia. It includes a post-start hook that automatically pulls the specific uncensored models requested (Dolphin, DeepSeek-Coder, DarkBERT).1
* Open WebUI: The script injects a custom user.css file into the container volume to apply the "Cyberpunk" theme (neon colors, high contrast).20 It also pre-populates the config.json with the "Renegade" system prompt, ensuring the AI wakes up with the correct attitude.
* Unraid MCP Integration: The script configures the Open WebUI "Tools" connection settings to point to the Unraid MCP server (http://192.168.1.222:6970), enabling the "Brain" to immediately start querying the "Brawn's" status.
9.4 Phase 4: The "Sentry" Modules
* Network Scanning: The script uses apt to install nmap and metasploit-framework directly on the host OS. It generates a Python script, vulnbot.py, which is placed in the user's home directory. This script acts as a wrapper for nmap, allowing the AI (via Open WebUI's "Code Interpreter" or tool use) to execute network scans safely within the local subnet (192.168.1.0/24).1
* Paywall Bypass: The 13ft Ladder container is deployed. The script configures the Crawl4AI service to route HTTP requests through this container when it encounters HTTP 403 (Forbidden) or specific "Paywall" distinct markers in HTML, ensuring high-success-rate ingestion.
